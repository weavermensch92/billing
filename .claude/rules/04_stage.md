# 04_Stage — 도입 단계 분기 규칙

> Gridge AIMSP의 **점진적 도입** 철학. 고객이 AI를 한 번에 전면 도입하지 않고 단계적으로 확장.
> 각 Stage에서 UI/AI 실행 범위/HITL 작동이 어떻게 달라지는지 명시.
> 규칙 ID: G-060~G-079.

---

## 1. Stage 정의 (G-060)

### 4단계 철학

| Stage | 명칭 | AI 코드 | 주 가치 | 전환 기준 |
|---|---|---|---|---|
| **0** | 모니터링 | ❌ 없음 | 적합화 규칙 세팅 | 초기 진입 |
| **1** | AI 보조 | 제안만 | AI 제안 + 개발자 구현 | 규칙 50%+ 확정 |
| **2** | AI 협업 | 모듈별 | 특정 모듈은 AI가 작성 | 특정 모듈 적합화 70%+ |
| **3** | AI 주도 | 풀 파이프라인 | R1~R7 → E0~E5 전체 | 적합화 점수 80%+ |

### G-060-01 — "Stage 0에서도 적합화는 동작"

**이것이 제품 핵심 포지셔닝.** AI 코드 없이 규칙 세팅만으로도 가치 있다는 증거.

- Stage 0 구축 기간: 2~4주
- Stage 0 고객이 얻는 것: 기존 코드베이스 분석 리포트 (62개 패턴 감지 + 네트워크 추천 14건 수준)
- **Stage 0만 하고 영영 상위 전환 안 해도 OK** (리스크 제로 포지셔닝)

### G-060-02 — 단계 건너뛰기 금지

- 0 → 2 ❌ (1 경유 필수)
- 1 → 3 ❌ (2 경유 필수)
- 상위 → 하위 전환은 가능 (적합화 점수 하락 시)

## 2. 네비게이션 분기 (G-061)

각 Stage에서 노출되는 상단 탭 목록:

| Stage | 노출 탭 |
|---|---|
| 0 | `[적합화] [설정]` |
| 1 | `[적합화] [칸반] [로그] [보고서] [설정]` |
| 2 | `[적합화] [칸반] [파이프라인] [로그] [산출물] [보고서] [설정]` |
| 3 | `[적합화] [칸반] [파이프라인] [로그] [산출물] [보고서] [운영] [설정]` |

**고정 원칙:**
- `적합화` 탭은 **항상 1번 위치** (Stage 무관, G-060-01 정합)
- `설정` 탭은 **항상 마지막**
- 상위 Stage에서 추가된 탭이 하위 Stage로 전환 시 사라짐 (데이터는 유지, 접근만 숨김)

### G-061-01 — OA 네비게이션 추가

Org Admin 로그인 시 추가 탭 1개:

| Stage | 추가 |
|---|---|
| 전체 Stage | `[조직 관리]` — OA만 노출 |

## 3. 칸반 분기 (G-062)

| Stage | 칸반 | 컬럼 | 특이사항 |
|---|---|---|---|
| **0** | **미표시** | — | 라우터 접근 시 `/adapt` 리다이렉트 |
| **1** | 표시 | 3컬럼 | `[제안] [검토] [반영]` |
| **2** | 표시 | 6컬럼 | 표준 6컬럼. **`agent` 필드 있는 아이템만 표시** |
| **3** | 표시 | 6컬럼 | 전체 아이템 표시 |

**표준 6컬럼 (Stage 2+):** `BACKLOG / SPEC / DEV / HITL / REVIEW / DONE`

### G-062-01 — Stage 2의 "AI 모듈만 표시" 의미

Stage 2는 **일부 모듈만 AI에게 위임**. 나머지 모듈은 수동 개발.
- `item.agent` 필드가 있으면 AI가 담당 → 칸반에 표시
- 없으면 수동 개발 중 → 칸반 숨김 (별도 tool 또는 외부 이슈 트래커 사용)

### G-062-02 — Stage 분기 위반 감지

```
❌ Stage 0 프로젝트에서 /kanban 라우트 접근 가능
❌ Stage 1 칸반에 6컬럼 표시
❌ Stage 2 칸반에 agent 필드 없는 아이템 표시
```

감지 시 Conflict 발동 (`91 § 1 조건 4` 정합성 위반).

상세: `products/wiring/rules/kanban.md § PW-008-01`.

## 4. 파이프라인 분기 (G-063)

| Stage | 파이프라인 | 표시 범위 |
|---|---|---|
| 0, 1 | ❌ 미표시 | — |
| 2 | ✅ 표시 | AI 모듈만 (agent 노드 + 해당 사람 노드) |
| 3 | ✅ 표시 | 전체 (모든 AI + 모든 사람 + 하네스) |

Stage 2에서는 **일부 노드만 활성화.** 나머지는 회색 idle 상태.

상세: `products/wiring/rules/pipeline_view.md` (PW-002~005).

## 5. 로그 분기 (G-064)

| Stage | 로그 내용 |
|---|---|
| 0 | 적합화 분석 로그만 (코드베이스 감지, 규칙 확정, 온톨로지 추천) |
| 1+ | 0 + AI 활동 로그 (6유형: 이관📤/실행⚙️/결정🎯/검증🔍/HITL⚡/완료✓) |

Stage 0 로그는 "제품이 살아있다"는 증거 — 고객이 접속 때마다 뭔가 진행되고 있음을 보여줌.

## 6. HITL 분기 (G-065)

| Stage | HITL 작동 |
|---|---|
| 0 | **작동함.** 규칙 확인/수정만 (코드베이스 분석 결과 기반) |
| 1 | 0 + AI 제안 검토 |
| 2 | 1 + 모듈별 AI 결과 검증 |
| 3 | 2 + 풀 파이프라인 승인 |

**Stage 0에서 HITL 작동하는 이유:** 적합화가 제품 핵심이기 때문 (G-060-01).

상세: `06_hitl.md § 6`.

## 7. 보고서 분기 (G-066)

| Stage | 보고서 내용 |
|---|---|
| 0 | 적합화 현황 (점수 추이, 카테고리별 규칙 수), 주간 결정 수 |
| 1 | 0 + 주간 진행 (아이템 이동 수), AI 제안 수락률 |
| 2 | 1 + 토큰 사용량 (모드별 분기), AI 코드 수정률 |
| 3 | 2 + ROI 계산기 (수동 vs AI 개발 비교) |

## 8. 운영 탭 분기 (G-067)

**Stage 3 전용** 탭. 하위 Stage에서는 탭 자체가 보이지 않음.

포함 기능:
- 배포 현황 (CI/CD 연동: Jenkins / ArgoCD)
- 이슈 피드 + 피드백 루프
- 장애 대응 (롤백, 회고)

---

## 9. Stage 전환 (G-068)

### 9.1 누가 전환할 수 있나

| 전환 방향 | 권한 |
|---|---|
| Stage 상승 | L2 PM or L3 기술 리드 (프로젝트 단위) |
| Stage 하락 | L2 PM or L3 (경고 후 진행) |
| 조직 전체 Stage 강제 | OA |

### 9.2 Stage 상승 시 자동 동작

```
N → N+1 전환 요청
  ↓
적합화 점수 확인 (기준 미달 시 경고)
  ↓
하네스 AI 재배정 (Stage에 맞는 에이전트 활성화)
  ↓
네비게이션 탭 갱신 (WebSocket)
  ↓
감사 로그 기록 (G-141)
  ↓
해당 프로젝트 전 사용자에게 알림
```

### 9.3 Stage별 적합화 점수 권장 기준

| Stage | 권장 적합화 점수 |
|---|---|
| 0 → 1 | 50%+ (기본 규칙 세팅 완료) |
| 1 → 2 | 70%+ (AI 제안 수락률 60%+) |
| 2 → 3 | 80%+ (AI 코드 수정률 20% 이하) |

**권장일 뿐 강제 아님.** 기준 미달 시 경고만 + OA 승인으로 진행 가능.

### 9.4 Stage 하락 트리거

자동 하락은 **없음**. 사용자가 명시적으로 선택.

수동 하락 권장 조건:
- 적합화 점수 30%+ 하락 (기술 스택 전면 변경 등)
- AI 코드 수정률 40%+ 상승
- Conflict 발생률 급증

## 10. Stage별 AI 실행 범위 (G-070)

| Stage | AI 실행 권한 |
|---|---|
| 0 | 코드베이스 분석, 규칙 추출, 온톨로지 추천만 |
| 1 | 0 + 코드 스니펫 제안 (사람이 직접 적용) |
| 2 | 1 + 모듈별 풀 구현 (PR 자동 생성) |
| 3 | 2 + 전체 파이프라인 (R1~R7, E0~E5) + 자동 배포 (승인 필요) |

### G-070-01 — Stage별 Gate 차이

- Stage 0~1: T1 정적분석만 (코드 거의 없음)
- Stage 2: T1 + T2 테스트 + T3 규칙 준수
- Stage 3: T1~T4 전부 (보안/컴플라이언스 포함)

---

## 11. Stage별 비용 구조 (G-071)

| Stage | 주요 비용 |
|---|---|
| 0 | 플랫폼 라이선스만 (AI 호출 미미) |
| 1 | 라이선스 + AI 제안 토큰 (제안만이라 적음) |
| 2 | 라이선스 + 모듈별 AI 구현 토큰 |
| 3 | 라이선스 + 풀 파이프라인 AI 토큰 |

### G-071-01 — 모드별 비용 표시 (Stage 교차)

| Stage | Mode A | Mode B | Mode C |
|---|---|---|---|
| 0 | 월 라이선스만 | 월 라이선스만 | 월 라이선스만 |
| 1~3 | 토큰 잔액 + 충전 | 비용 미표시 (고객 인프라) | USD breakdown |

상세: `05_infra_mode.md § 5` (모드별 비용 표시).

## 12. Org 내 여러 프로젝트의 Stage 혼재 (G-072)

한 조직(Org) 내에서 프로젝트마다 Stage 다를 수 있음:

```
Organization 코레일
├── Project 쇼핑몰 리뉴얼 (Stage 3)
├── Project 재고관리 API (Stage 2)
├── Project 정산 시스템 (Stage 1)
└── Project 배송 추적 (Stage 0)
```

### G-072-01 — 조직 대시보드 표시

OA 대시보드에서 프로젝트별 Stage 배지 표시. 팀별 집계는 가중 평균.

## 13. Stage + 위계 교차 (G-073)

| 위계 | Stage 0 | Stage 1 | Stage 2 | Stage 3 |
|---|---|---|---|---|
| OA | 조직 규칙 세팅 | 조직 규칙 + 파일럿 관리 | 조직 규칙 + 크로스 프로젝트 비용 | 전 기능 |
| L1 | 적합화 진행률 | 1 + 주간 요약 | 2 + 비용 추이 | 3 + ROI |
| L2 PM | 🔶 비즈 결정 | 1 + 칸반 관리 | 2 + 산출물 검토 | 3 + 운영 |
| L3 기술 리드 | 🔷 기술 결정 + 코드 패턴 | 1 + AI 제안 검토 | 2 + 하네스 재설계 | 3 + 전 파이프라인 |
| L4 개발자 | 코드 패턴 제출 | 1 + AI 제안 수용/거부 | 2 + 모듈 리뷰 | 3 + 풀 리뷰 |

## 14. 자동 검증 체크리스트

체인 실행 중 Stage 분기 누락 감지 시 Conflict 자동 발동:

- [ ] Stage 0 프로젝트에서 칸반/파이프라인/운영 탭 접근 가능?
- [ ] Stage 1 칸반이 6컬럼으로 표시?
- [ ] Stage 2 칸반에 `agent` 필드 없는 아이템 표시?
- [ ] Stage 3 전용 운영 탭이 Stage 2 이하에 노출?
- [ ] Stage 0에서 적합화 탭이 비활성화? (G-060-01 위반)
- [ ] Stage 전환 시 감사 로그 기록 누락?
- [ ] Stage 단계 건너뛰기 허용? (G-060-02)
