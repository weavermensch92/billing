# 90_Execution Chain — 실행 체인 7종

> Claude Code가 사용자 요청을 받았을 때 **자동으로 따르는 실행 순서**.
> CLAUDE.md § 2.2 에서 키워드 감지로 체인을 선택한 후, 이 문서의 해당 섹션을 그대로 실행.

**공통 원칙:**
- 각 체인은 **사전 로드 → 탐색 → 실행 → 검증 → 기록 → 산출물** 6단계.
- 탐색 단계에서 INDEX.md 우선. 전체 디렉토리 스캔 금지.
- 실행 단계 중 Conflict 감지 시 즉시 `99_protocol.md § Conflict` 로 전환.
- 모든 체인 종료 시 `/gz-self-improve` 자동 제안.

### 파일 부재 시 공통 Fallback (모든 체인 적용)

체인이 요구하는 파일(사전 로드, 탐색 대상, 검증 레퍼런스)이 부재할 때:

| 상황 | 동작 |
|---|---|
| 지시된 파일 1개 부재 | ALWAYS_LOAD에 있는 관련 규칙으로 우회 (00_index에서 가장 가까운 상위 범주 확인) |
| 지시된 파일 과반수 부재 | **체인 실행 중단** + 사용자에게 "본문 파일 부재" 고지 + 계속 여부 확인 |
| 규칙 ID는 등록됐는데 본문 부재 | 00_index의 요약만으로 진행하되, **자가 리뷰 단계에서 "본문 미검증" 경고 표기** |
| `.samples/` 빈 디렉토리 또는 부재 | 스킵 + 기존 코드 git grep으로 대체. 부재 자체는 Conflict 대상 아님 |
| `.context/config.yml` / `session.yml` 부재 | 4축을 사용자에게 묻되, 답변 받으면 **자동 생성** (99 § 부록 참조) |

**자가 리뷰 경고 형식:**
```
⚠ 본문 미검증
  참조한 규칙: PW-007 (위계별 탭 노출 필터)
  참조 경로: products/wiring/rules/adapt_tab.md (부재)
  사용한 대체 정보: 03_hierarchy.md § 3.2 표
  신뢰도: 중 (규칙 존재는 확실하나 상세 조건 미확인)
```

## F. Feature Implementation (신규 기능 구현)

### 트리거
"구현", "만들어", "추가", "~화면 개발", "~페이지 만들기", "새 기능"

### 사전 로드 (1단계)
```
1. CLAUDE.md 4축 확정
2. 해당 제품 products/{p}/CLAUDE.md
3. 연관 규칙: rules/07_coding_standard.md + 해당 제품 rules/
4. schemas/INDEX.md (DB 접근 예상 시)
```

### 탐색 (2단계)
```
2.1 레퍼런스 샘플 탐색
     - .samples/{도메인}/ 경로 존재 시 우선 읽기
     - 유사 기능 구현된 기존 파일 git grep
2.2 관련 스키마 탐색
     - schemas/{project}/INDEX.md 에서 필요 테이블만 특정
     - 전체 테이블 스캔 금지
2.3 기존 패턴 탐색
     - server actions / services 기존 패턴
     - 컴포넌트 레이아웃 기존 패턴
2.4 규칙 ID 수집
     - 00_index.md 자동 로드 트리거 섹션 기반
     - 예: "포인트" 언급 → D-025 자동 로드
```

### 실행 (3단계)
```
3.1 Todo 작성 → 사용자 확인
3.2 types.ts 생성 (있으면 수정)
3.3 server action 생성 (DB 접근 시)
3.4 컴포넌트 생성/리팩토링 (하향식)
3.5 페이지 조립
     - 각 파일 생성 후 즉시 타입 체크
     - 실행 실패 시 즉시 중단 (재시도 3회 제한 → Conflict)
```

### 검증 (4단계)
```
4.1 코딩 표준 자가 점검 (G-120~G-125)
4.2 해당 제품 규칙 위반 검토
4.3 분기 필요성 판정 후 누락 검토 (아래 표 참조)
4.4 타입 에러 0건 확인
```

**4.3 분기 필요성 판정 매트릭스:**

| 분기 축 | 이 기능에 필요한 조건 | 필요한데 누락 시 |
|---|---|---|
| **위계 (target)** | UI가 사용자 역할에 따라 다르게 보여야 하는가? 권한 체크가 있는가? | **Conflict 발동** (G-052 위반) |
| **모드 (A/B/C)** | 비용 표시 / 세션 배지 / 데이터 반출 관련 기능인가? | **Conflict 발동** (G-080~085 위반) |
| **Stage (0~3)** | 탭·메뉴·네비게이션의 노출 범위가 달라지는가? 칸반·파이프라인·보고서 화면인가? | **Conflict 발동** (G-062 위반) |

**판정 기준 (AND 조건 아님, 어느 하나만 해당해도 필요):**
- 위계 필요: 권한 확인 / 서버 필터링 / 역할별 UI / SSO 매핑 관련
- 모드 필요: 토큰·자원·USD 표시 / 고객 API 키 사용 / 온프레 데이터 경계
- Stage 필요: Stage별 네비 노출 / AI 실행 범위 / 적합화와 AI 협업 경계

**불필요 판정 예시 (분기 없어도 OK):**
- 로그인 화면, 에러 페이지, 404: 전 축 불필요
- 읽기 전용 통계 표시: 위계는 필요, 모드·Stage 보통 불필요
- 순수 데이터 변환 유틸: 전 축 불필요

### 기록 (5단계)
```
5.1 세션 중 새로 발견된 패턴 → .claude/issue/knowledge-*.md 초안
5.2 3회 이상 동일 수정 → /gz-pattern 자동 제안
5.3 commit 메시지 포맷: 
    "feat({scope}): {목적}
    Refs: {규칙 ID list}
    Chain: F"
```

### 산출물 (6단계)
- 신규/수정 파일 목록
- 적용된 규칙 ID 목록
- 분기 처리된 축 (모드/Stage/위계)
- PR draft (branch: `feat/{slug}`)

### 예시 매핑 (denatoz-ax Image 8 참조)
사용자: "Wiring 칸반 페이지에 HITL 필터 기능을 추가해줘"
→ F 체인 발동
→ 사전: Wiring CLAUDE.md + PW-008(칸반) + G-044(역할 라우팅)
→ 탐색: 기존 칸반 컴포넌트 git grep → 필터 바 기존 패턴 확인
→ 실행: types → action → filter-bar → kanban 통합
→ 검증: L2/L3 위계별 필터 노출 차이 확인 (PW-007)
→ 기록: knowledge-*_kanban-filter-pattern.md

```
2. 리팩토링 전/후 동일 입력 → 동일 출력 보장
3. 복잡도 감소 증명 (라인 수, 함수 길이, 중첩 깊이)
```

### 기록

## S. Schema Change (DB 스키마 변경)

### 트리거
"스키마", "테이블 추가", "컬럼 변경", "엔티티 수정", "마이그레이션 스크립트"

### 사전 로드 (절대 순서)
```
1. schemas/{project}/INDEX.md 로 대상 테이블 특정
2. schemas/{project}/tables/{table}.md DDL 확인
3. rules/02_architecture.md
4. PL-xxx (LucaPus) 또는 PA-xxx (AiOPS) DB 규칙
5. G-141 (감사 로그), G-144 (고객 격리)
```

### 탐색
```
1. 외래키 참조 관계 전수 확인 (FK는 논리적만 — G-xxx 설계 원칙)
2. 감사 대상 여부 (_info/_log/_code 네이밍 규약)
3. JSON 컬럼 설계 재검토 (JSONB 사용 + GIN 인덱스 필요 여부)
```

### 실행 (엄격)
```
1. ❗ 컬럼 추가/수정/삭제는 Claude Code가 직접 마이그레이션 수행 금지
2. DDL 변경 사양서만 작성 → 사용자(개발자)에게 전달
3. 사용자가 실제 마이그레이션 후 스키마 재동기화
4. 타입/엔티티 코드 업데이트는 Claude Code 수행 가능
```

### 검증
```
1. 필수 공통 컬럼 확인 (idx BIGSERIAL PK, created_at, updated_at)
2. 네이밍 규약: idx/_idx(FK)/_at/_date/is_/_image/_file/_path/_json
3. DELETE 금지 — is_del 플래그로 논리 삭제 (MUST)
4. 쿼리 내 연산 금지 (계산은 앱 레벨)
5. 컬럼 단일 책임 (상태+타입 혼합 금지)
```

### 기록
- DDL 변경 사양서: `.claude/issue/schema-YYYY-MM-DD_<table>.md`
- 감사 대상 변경 시 조직 규칙(G-141) 업데이트 필요 → Feedback 자동 기록

### 탐색
```
1. 증상 재현 — 사용자로부터 최소 재현 단계 확보
     → 불가능 시: 스택 트레이스/로그 요청
2. 관련 파일 git blame 으로 최근 변경자 / 시점 확인
3. 테스트 존재 확인 → 결함 재현 테스트 먼저 작성
```

### 실행
```
1. 가설 → 검증 → 가설 수정 사이클
2. 같은 가설 3회 실패 → 즉시 중단 (§91 Conflict: 반복 버그)
3. 임시 우회 (hotfix) vs 근본 수정 (root cause) 판단 → 사용자 확인
```

### 검증
```
1. 결함 재현 테스트가 이제 통과하는지
2. 주변 테스트 회귀 없는지
3. 같은 원인이 다른 곳에도 있는지 전수 검사 (같은 패턴 grep)
```

### 기록
- 원인/해결 패턴 → `.claude/issue/knowledge-*_{bug-summary}.md`
- 같은 패턴 재등장 → 자동 Conflict 트리거 조건 업데이트

## 체인 외 (비-체인 요청)

"이거 어떻게 해?", "이 개념 설명해줘", "A와 B 중 뭐가 나아?" 같은 질의응답형 요청은 **체인 미발동.**
그냥 해당 영역 규칙 로드 + 답변. 실행 체인은 "코드를 생성/변경" 할 때만 발동.
