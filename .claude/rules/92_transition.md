# 92_Transition — 전환·핸드오프·에스컬레이션 규칙

> "**지금 이 규칙을 계속 적용할 것인가, 다른 규칙으로 이동할 것인가**"를 결정하는 메타 규칙.
> 체인 내부의 실행 순서는 `90_execution_chain.md` 가, 충돌 감지는 `99_protocol.md` 가 담당.
> 이 문서는 **경계 넘기**를 다룬다: 체인 ↔ 체인, AI ↔ 사람, 세션 ↔ 세션.

---

## 1. 핸드오프 매트릭스 — AI ↔ 사람

### G-180 — 판단 주체 결정 트리

작업 중 결정이 필요한 상황이 오면 Claude Code는 **자동으로 판단 주체를 분기**한다.

| 결정 유형 | 판단 주체 | 근거 |
|---|---|---|
| 비즈니스 정책 (환불/쿠폰/가격) | **L2 PM** | G-104 비즈니스 결정 라우팅 |
| 아키텍처 선택 (패턴/락 전략/분리 구조) | **L3 기술 리드** | G-103 기술 결정 라우팅 |
| 코드 패턴 승격 (3회 반복 감지) | **L3 기술 리드** | G-044 적합화 주체 |
| 신규 규칙 ID 할당 | **L3 기술 리드** | `98_governance § 6` |
| 보안/컴플라이언스 (PII/감사/Mode B) | **L3 + OA 공동** | G-140~179 |
| DB 마이그레이션 실행 | **L4 개발자** | `90 § S` Schema 체인 |
| 외부 API 키·시크릿 발급 | **L3 기술 리드** | G-144 |
| 프로덕션 배포 결정 | **L3 + OA 공동** | `93 § 6` |
| 조직 규칙 추가 (MUST) | **OA 단독** | G-042 |
| 제품 간 연동 계약 확정 | **양쪽 제품 L3 공동** | `90 § I` |

### G-181 — AI가 단독 결정 가능한 범위

위 매트릭스에 없는 것 중 AI가 단독 수행 가능한 것:

- SHOULD/MAY 규칙 해석과 적용
- 코딩 표준 준수 (G-120~G-125)
- 기존 패턴 재사용 (규칙 ID로 명시된 경우)
- 타입 정의, 유닛 테스트 작성
- 리팩토링 (공개 API 변경 없음)
- 문서 업데이트 (기존 규칙 참조)

**AI 단독 범위를 벗어나면 즉시 §G-180 매트릭스로 돌아가 주체 확인.**

### G-182 — 핸드오프 시 전달 형식

AI → 사람 인계 시 아래 형식 필수:

```
[핸드오프] <결정 유형>

현재 상황:
  <한 줄 요약>

배경:
  <관련 규칙 ID + 2~3줄 맥락>

옵션:
  [A] <선택지 + 장단점>
  [B] <선택지 + 장단점>
  [C] 다른 방향 (직접 입력)

AI 권장: <A/B/C 중 하나 + 근거 + 신뢰도>

결정 후 자동 실행: <선택에 따라 어떤 체인이 이어지는지>
```

## 2. 체인 간 전환 규칙

### G-183 — 체인 전환 트리거

한 체인 실행 중 아래 신호를 감지하면 **사용자 확인 후** 다음 체인으로 이동:

| 현재 체인 | 전환 트리거 | 다음 체인 |
|---|---|---|
| F (Feature) | 기존 코드 중복 3회 감지 | R (Refactoring) |
| F | 타 제품 인터페이스 변경 필요 감지 | I (Integration) |
| F | 스키마 변경 필요 감지 | S (Schema) → 복귀 후 F |
| D (Debug) | 수정 완료 + 재발 방지 테스트 작성 요청 | V (Verification) |
| D | 원인이 다른 모듈의 버그로 밝혀짐 | D (신규 세션) |
| S (Schema) | DDL 적용 완료 | F (관련 기능 업데이트) |
| R (Refactor) | 리팩토링 중 버그 발견 | D (우선) |
| V (Verify) | 불일치 발견 → 수정 필요 | F 또는 D |
| M (Migration) | 범위 확장 필요 | 중단 후 사용자 확인 |
| I (Integration) | 양쪽 제품 중 한쪽이 스키마 변경 필요 | S |

### G-184 — 체인 복귀 규칙

전환 후 원 체인으로 돌아와야 하는 경우:

- F → S → F : 스키마 변경 후 기능 구현 재개 (**자동 복귀**)
- R → D → R : 리팩토링 중 버그 수정 후 재개 (**자동 복귀**)
- F → I → F : 연동 계약 확정 후 기능 구현 재개 (**자동 복귀**)

**자동 복귀 시에도 원래 체인의 4축(제품/모드/위계/Stage)은 유지.** 변경되었으면 `/gz-scope` 재확인.

### G-185 — 체인 중단 규칙

중단 후 재개 불가 상황:

- 범위가 MVP 1개 스프린트를 초과 → **M 체인으로 승격하고 사용자 승인 재획득**
- 중간 산출물이 10개 파일 초과 + 미검증 → **강제 검증 단계 삽입**
- 초기 4축이 틀렸다고 판명 → **현재 체인 중단 + 롤백 + 4축 재확정**

## 3. 에스컬레이션 계단

### G-186 — 실패 단계별 대응

동일 시도의 실패 횟수에 따른 자동 대응:

| 시도 | 실패 시 동작 | 담당 |
|---|---|---|
| 1회 실패 | 접근법 변경, 재시도 | AI 단독 |
| 2회 실패 | **접근법 근본 재고** + 가설 재수립 | AI + 사용자 고지 |
| 3회 실패 | **Conflict 자동 발동** (`91 § 1`) | 작업 중단 + 사용자 결정 |
| 3회 후 Conflict 미해결 | **L3 기술 리드 호출** | GitHub Issue 자동 생성 |
| L3 부재 시 | 48시간 대기 → OA 에스컬레이션 | `.claude/issue/blocked-*.md` |

### G-187 — 규칙 위반 시도 감지

사용자 지시가 MUST 규칙을 위반하려 할 때:

1. **즉시 중단** + 위반 의심 고지
2. 해당 규칙 ID 본문 인용
3. 3가지 옵션 제시:
   - [A] 규칙 준수하는 대안 실행
   - [B] 규칙 예외 신청 (L3 승인 필요 → `/gz-feedback` 자동)
   - [C] 규칙 자체 수정 제안 (OA + L3 공동 승인 → `/gz-feedback`)
4. 사용자가 [B] 또는 [C] 선택 시 별도 세션으로 분리 (현재 작업은 대기)

### G-188 — 정합성 위반 감지 (LucaPus 7원칙)

`CLAUDE.md § 6` 의 7원칙은 **예외 없음.** 위반 시도 감지 시:

1. **Conflict 자동 기록** (§ G-186 3회 실패와 별개로 즉시 발동)
2. 해당 위반이 요구되는 이유가 **신규 제품 요구사항**인 경우 → OA에게 에스컬레이션
3. 원칙 자체를 바꿔야 한다면 `98_governance § 8` 수정 프로토콜

## 4. 컨텍스트 포화 대응

### G-190 — 세션 한계 감지

아래 지표 중 하나라도 임계치 초과 시 사용자에게 **세션 분할 제안**:

| 지표 | 주의 | 경고 | 강제 분할 |
|---|---|---|---|
| 로드한 파일 수 | 30개 | 50개 | 70개 |
| 대화 토큰 | 60% | 75% | 90% |
| 동일 파일 재read 횟수 | 3회 | 5회 | 7회 |
| 체인 실행 시간 | 30분 | 1시간 | 2시간 |
| 미저장 변경 파일 | 10개 | 20개 | 30개 |

### G-191 — 세션 분할 절차

경고 이상 단계에서:

1. 진행 중인 체인 **단계 경계에서** 중단 (파일 중간 절대 금지)
2. `/gz-self-improve` 자동 실행 → Knowledge 추출
3. 현재 4축 + 진행 상태 → `.context/session.yml` 저장
4. 중단 지점 명시 → `.claude/issue/resume-YYYY-MM-DD_HH-MM.md`
5. 사용자에게 재개 안내

### G-192 — 세션 재개 절차

새 세션 시작 시 아래 순서 **자동 실행**:

1. `.context/session.yml` 로드 → 이전 4축 복원 + 사용자 확인
2. `.claude/issue/resume-*.md` 존재 시 읽기 → 중단 지점 복원
3. 이전 세션의 `knowledge-*.md` 전량 스캔 → 관련 내용 컨텍스트 주입
4. 이전 세션에서 해결 안 된 `conflicts-*.md` 있으면 **최우선 처리**
5. 이전 진행 상태 보고 + "계속할까요, 재조정할까요?" 질문

## 5. 작업 우선순위 경쟁 해결

### G-193 — 동시 작업 요청 대응

사용자가 이미 실행 중인 체인이 있는 상태에서 새 작업을 요청하면:

```
현재 F 체인 진행 중 (Wiring 칸반 필터, 60% 완료)

옵션:
[1] 현재 작업 완료 후 새 작업 착수 (추천)
[2] 현재 작업 체크포인트 저장 후 새 작업 즉시 시작
[3] 새 요청을 현재 체인에 병합 (같은 컨텍스트 내 추가 기능)
[4] 새 요청 기각
```

**병합([3]) 가능 조건:** 동일 제품 + 동일 4축 + 동일 체인 유형.

### G-194 — 긴급 요청 끼어들기

"긴급", "프로덕션 다운", "지금 당장" 등 긴급 신호 감지 시:

1. 현재 작업 **즉시 체크포인트 저장**
2. 새 작업을 D 체인으로 강제 진입
3. 긴급 대응 후 이전 작업 자동 재개 (자리 유지)

단, 긴급 주장이 `90 § D` D 체인 트리거에 부합하는지 AI가 1차 판단. 의심 시 사용자 재확인.

## 6. 세션 종료 규칙

### G-195 — 종료 의사 감지 키워드

"오늘은 여기까지", "내일 이어서", "저장해줘", "마무리", "끝", "쉬자" 등 감지 시:

1. 현재 체인 **안전 중단 지점까지만** 진행
2. `/gz-self-feedback` 자동 실행 → Knowledge + Feedback 동시 생성
3. 미완료 작업 요약 → `.claude/issue/resume-*.md`
4. PR draft 상태 정리 (WIP 태그 부착)
5. 다음 세션 재개 안내 메시지

### G-196 — 작업 완료 감지

작업이 **실질적으로 완료**되었다고 판단되는 조건:

- 체인의 산출물(6단계) 전부 생성됨
- 모든 검증(4단계) 통과
- 타입 에러 0건, 테스트 전량 통과
- 사용자 명시 승인

→ `/gz-self-improve` 자동 실행 → PR 생성 → 체인 정리.

## 7. 장기 미접속 대응

### G-197 — 7일 이상 미접속 후 재개

이전 세션이 7일 이상 지난 후 재개되면:

1. **전체 룰북 재스캔** (규칙이 업데이트되었을 수 있음)
2. `npm outdated` 또는 하네스 버전 확인 → 업그레이드 제안
3. 이전 세션의 4축이 여전히 유효한지 재확인 (제품/모드/위계/Stage 중 변경된 것 없는지)
4. 이전 knowledge 중 이미 `rules/` 에 반영된 것은 제외하고 필요한 것만 컨텍스트 주입

### G-198 — 30일 이상 미접속

이전 세션이 30일 이상 지나면 **이전 세션 컨텍스트는 신뢰하지 않는다.**

- 4축 전면 재확정
- 이전 knowledge는 참고로만 사용 (규칙 레벨로 승격된 것만 신뢰)
- 프로젝트 상태 완전 재진단 권장

## 8. 전환 이벤트 자동 기록

모든 전환(핸드오프/체인 변경/에스컬레이션/세션 분할)은 자동 로그 대상:

```
.claude/issue/transition-YYYY-MM-DD_HH-MM_<이벤트>.md
```

내용:
- 전환 전 상태 (체인/4축/진행률)
- 전환 이유
- 전환 후 상태
- 책임자 (AI / L3 / OA 등)

전환 로그는 `/gz-send-issue` 전송 시 **별도 라벨** `transition` 할당. 분기별 회고 자료.

## 9. 이 문서가 바라는 효과

Claude Code가 작업을 진행하면서 아래 판단을 **자동으로** 수행하는 것:

- "이 결정은 내가 해도 되나, 사람이 해야 하나?"
- "지금 체인을 유지해야 하나, 다른 체인으로 넘어가야 하나?"
- "이 실패는 재시도인가, Conflict인가, 에스컬레이션인가?"
- "세션이 너무 길어지고 있는 건 아닌가?"
- "사용자가 작업을 종료하려는 신호를 보내고 있지 않은가?"

이 판단이 내재화되면 사용자는 **"개발 흐름"을 지시하지 않고 "목적"만 지시**해도 작업이 스스로 흐른다.
