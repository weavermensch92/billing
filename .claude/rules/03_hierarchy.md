# 03_Hierarchy — 위계·권한·라우팅

> 그릿지 AIMSP는 **6단 위계** × **3계층 조직** × **3제품별 권한 체계**의 곱으로 정의된다.
> 모든 기능이 이 매트릭스에 의존. 규칙 ID: G-040~G-059.

---

## 1. 위계 정의 (6단)

### G-040 — 위계 코드와 접점

| 위계 | 코드 | 범위 | 주 접점 | 핵심 책임 |
|---|---|---|---|---|
| 슈퍼 어드민 | `super` | 전 고객사 | 내부 관리 | 고객사 생성, 인프라 모드 할당 |
| Org Admin | `org_admin` | 조직 전체 | 웹 (조직 관리) | 조직 규칙, 팀 관리, SSO, 감사 |
| L1 경영진 | `executive` | 팀/프로젝트 | 웹 (대시보드) | 조직 전체 가시화, 비용·보고서 열람 |
| L2 PM | `pm` | 프로젝트 | 웹 (적합화/칸반) | 비즈니스 결정, 프로젝트 진행 관리 |
| L3 기술 리드 | `tech_lead` | 프로젝트 | CLI + 웹 | 기술 결정, 규칙 승격, 하네스 재설계 |
| L4 개발자 | `developer` | 할당 작업 | CLI(주) + 웹 | 코드 리뷰, 패턴 제출 |

**위계 사용 원칙:**
- 주 접점이 `CLI(주) + 웹`인 위계(L3/L4)는 **CLI 기능 동등**이 기본. 웹에서만 가능한 기능은 명시되어야 함
- 슈퍼 어드민은 고객사 밖의 존재. 고객 조직 내부 권한 체계와 분리
- L1은 **읽기 전용.** 실행 권한 없음. 가시화만
- OA 는 조직 전체를 관리하지만 **프로젝트 내 실행**은 하지 않음 (설계상 PM/L3 영역)

### G-041 — 조직 계층

```
Organization (OA가 관리)
  ├── Team A
  │     ├── Project 1
  │     └── Project 2
  ├── Team B
  │     └── Project 3
  └── 조직 규칙 (MUST → 전 팀 강제)
```

- Organization: 고객사 1개 = Org 1개
- Team: 부서/제품 단위 (예: Platform팀, Backend팀)
- Project: 실제 개발 대상 레포/서비스
- 한 사람이 **여러 Team** 소속 가능 (L2~L4), 여러 Project 참여 가능
- OA는 한 Org당 복수 가능 (상호 견제)

## 2. 규칙 상속 (G-042, G-043)

### G-042 — 조직 MUST 하위 해제 금지

| 레벨 | 예시 | 하위 오버라이드 |
|---|---|---|
| **조직 MUST** | "BCrypt 해싱 필수", "감사 로그 1년+" | **절대 불가** (하위에서 해제 시도 시 차단) |
| 조직 SHOULD | "API 버저닝 권장" | 팀/프로젝트에서 MUST 승격 또는 MAY 완화 가능 |
| 팀 규칙 | "Backend팀 Facade 패턴" | 프로젝트에서 오버라이드 가능 |
| 프로젝트 규칙 | "이 프로젝트 Redis 캐시" | 이 프로젝트 한정 |

### G-043 — 상속 UI 표시

설정 > 개발 규칙에서 각 규칙 옆에 상속 출처 라벨:

- `[조직 🔒]` — OA가 확정, 해제 불가
- `[팀]` — 팀 리드가 확정, 프로젝트 오버라이드 가능
- `[프로젝트]` — 이 프로젝트 한정

라벨 없음 = 프로젝트 기본값.

**상속 충돌 감지:** 팀 간 동일 도메인에 다른 규칙 적용 시 **OA/L1에게만** 경고 노출 (→ PW-013 Org Admin 화면).

## 3. 적합화 큐 자동 라우팅 (G-044 — 핵심)

### 3.1 카드 유형별 담당자

| 카드 유형 | 아이콘 | 담당 위계 | 근거 |
|---|---|---|---|
| **비즈니스 결정** | 🔶 주황 | **L2 PM** | 정책/가격/UX 판단 |
| **기술 결정** | 🔷 파랑 | **L3 기술 리드** | 아키텍처/패턴/락 전략 |
| **코드 패턴 승격** | 🔶 주황 | **L3 기술 리드** (L4 제출 가능) | 반복 패턴 규칙화 |
| **온톨로지 추천** | 🔗 점선 파랑 | **L3 기술 리드** | 네트워크 통계 기반 |

### 3.2 위계별 적합화 탭 필터

**같은 화면, 다른 카드.** 로그인 역할에 따라 자동 필터:

| 위계 | 보이는 카드 | 예상 처리량 |
|---|---|---|
| L2 PM | 🔶 비즈니스만 | 전체의 ~10% |
| L3 기술 리드 | 🔷 기술 + 🔶 코드 패턴 + 🔗 온톨로지 | 전체의 ~70% |
| L4 개발자 | 본인이 제출한 패턴만 | 전체의 ~20% |
| OA/L1 | 조직 전체 통계만, 개별 카드 직접 처리 X | — |

### 3.3 라우팅 결정 알고리즘

에이전트(SSOT Master 등)가 HITL 노드를 발생시킬 때:

```
function routeDecision(node):
  if node.type == "policy_confirm":       return "L2"     # 기획서 정책
  if node.type == "tech_decision":        return "L3"     # 기술 선택
  if node.type == "code_pattern":         return "L3"     # 반복 감지
  if node.type == "ontology_recommend":   return "L3"     # 네트워크 추천
  if node.type == "rule_conflict":        return "OA"     # 조직 간 충돌
```

**원칙 (PRD 교정 원칙 1):** PM에게 기술 결정을 라우팅하지 않는다. 적합화의 주체는 개발자.

### 4.2 AiOPS 별도 권한 체계 (G-047)

AiOPS는 기업 AI 사용 모니터링 제품이라 **별도의 3단 권한**을 쓴다 (법적 이슈 대응):

| AiOPS 권한 | 범위 | 수정/삭제 |
|---|---|---|
| `super_admin` | org 전체 로그 열람, 계정 관리 | 불가 (변경 금지) |
| `admin` | 담당 팀(복수 가능) 로그 열람 | 불가 |
| `member` | 본인 로그만 열람 | 불가 |

**모든 AiOPS 권한은 수정/삭제 불가.** 로그는 immutable.

### G-048 — Wiring ↔ AiOPS 권한 매핑

Wiring+AiOPS 통합 운영 시 권한 매핑:

| Wiring 위계 | AiOPS 권한 | 추가 조건 |
|---|---|---|
| OA | `super_admin` | 자동 |
| L1 경영진 | `super_admin` | 기본값, OA가 `admin`으로 강등 가능 |
| L2 PM | `admin` | 담당 프로젝트의 팀만 |
| L3 기술 리드 | `admin` | 담당 팀만 |
| L4 개발자 | `member` | 본인만 |

**단독 판매 시:** AiOPS만 도입한 고객은 Wiring 위계 없음. AiOPS 3단 권한만 사용.

## 5. 특수 행위 권한 (G-049)

### 5.1 절대 권한 (슈퍼 어드민 전용)

- 고객사 생성/삭제
- 인프라 모드 A/B/C 할당 및 변경
- Mode B 온프레 엔드포인트 설정
- Mode A 고객사별 토큰 잔액 조정

### 5.2 공동 승인 필요 행위 (OA + L3)

- 조직 MUST 규칙 추가 (파급 효과 큼)
- 제품 간 연동 계약 확정 (I-xxx)
- 보안/컴플라이언스 관련 변경
- 프로덕션 배포 승인
- 룰북 자체(`rules/`) 수정

### 5.3 L3 단독 권한

- 하네스 재설계 요청 (모델 배정 변경 의뢰)
- 코드 패턴 규칙 승격 확정
- 기술 결정 확정 (🔷 카드)
- 데이터 내보내기 (프로젝트 범위)

### 5.4 절대 금지 행위 (위계 무관)

- 고객이 에이전트 모델 직접 변경 — **하네스 AI만 결정** (G-025)
- 감사 로그 수정/삭제 — immutable (G-141)
- 조직 MUST 규칙 하위 해제 (G-042)
- LucaPus 정합성 7원칙 위반 (`CLAUDE.md § 6`)

## 6. 오케스트레이션 뷰 스코프 (G-050)

파이프라인 탭의 **접근 권한별 노출 범위** (PW-002~005와 연동):

| 위계 | 보이는 것 | 안 보이는 것 |
|---|---|---|
| **OA / L1** | 조직 전체: 팀별 적합화 현황, 프로젝트 목록, 조직 규칙 적용 현황, **팀 간 규칙 충돌 감지** | 개별 프로젝트의 에이전트 실행 상세 |
| **L2 PM** | 프로젝트: 에이전트 실행 요약, 내 HITL 대기, 규칙 개수+추이, "기술 리드에게 위임됨" 표시 | 규칙 관계 그래프 상세, 기술 결정 세부 |
| **L3 기술 리드** | 프로젝트: 에이전트 실행 상세 + **규칙 관계 그래프**(requires/depends-on/triggers/serves) + 상속 경로 + 네트워크 통계 + 미설정 추천 | — (전부 노출) |
| **L4 개발자** | 할당 작업 + 관련 규칙 + 본인이 기여한 패턴 | 프로젝트 전체 규칙 그래프 |

**원칙:** 같은 URL, 다른 렌더링. 위계 정보는 클라이언트 조건 분기가 아닌 **서버 응답 레벨 필터링**.

## 7. SSO 역할 매핑 (G-046)

### 7.1 기본 매핑

IdP에서 받은 역할 → Wiring 위계 자동 매핑 규칙:

| IdP 역할 (예) | Wiring 위계 |
|---|---|
| `super-admin`, `org-owner` | OA |
| `executive`, `director`, `cto` | L1 |
| `product-manager`, `pm`, `product-owner` | L2 |
| `tech-lead`, `lead-engineer`, `principal` | L3 |
| `developer`, `engineer`, `member` | L4 |

### 7.2 지원 프로토콜

- SAML 2.0 (Okta, Azure AD, OneLogin)
- OIDC (Auth0, Keycloak, Google Workspace)
- SCIM 2.0 (사용자 프로비저닝 자동화 선택)

### 7.3 매핑 룰 우선순위

1. OA가 수동 지정한 매핑이 있으면 그것
2. 없으면 G-046 기본 매핑
3. 매칭 실패 시 → L4 개발자 (최소 권한)로 할당 + OA 알림

## 8. 위계 전환 규칙 (G-051)

### 8.1 전환 권한

| 전환 방향 | 가능한 주체 |
|---|---|
| 신규 사용자 초대 (L2/L3/L4) | OA, PM (프로젝트 한정) |
| 위계 승급 (L4→L3, L3→L2) | OA |
| 위계 강등 | OA |
| OA 추가 | 기존 OA (최소 1명 유지 필수) |
| OA 제거 | 기존 OA (자기 자신 제거 불가, 대체 OA 필요) |
| 슈퍼 어드민 지정 | 그릿지 내부 프로세스만 |

### 8.2 전환 시 영향

- **권한 변경은 즉시 적용**, 단 현재 진행 중 체인은 마무리 허용 (`92 § G-185`)
- 적합화 큐 대기 중인 카드 **재라우팅** (위계 맞는 새 담당자에게)
- SSO 연동 시 IdP 변경이 원천. Wiring 수동 변경은 OA 오버라이드로 기록
- 강등 시 해당 사용자가 승격한 규칙은 유지 (감사 기록)

### 8.3 전환 감사 대상

모든 위계 전환은 immutable 감사 로그 (G-141):

```
{ action: "권한 변경", actor: "관리자(OA)", target: "박주니어",
  before: "L4 개발자", after: "L3 기술 리드", at: "..." }
```

## 9. UI 노출 분기 원칙 (G-052)

### 9.1 기본 원칙

**서버에서 필터링한 데이터만 클라이언트에 보낸다.** 클라이언트 조건 분기로 숨기지 않는다.

```
❌ 나쁜 예:
  if (user.level === "L2") hideCard(techDecisionCard);

✅ 좋은 예:
  GET /api/adapt-queue?scope=auto
  → 서버가 user.level 확인 후 L2에게는 비즈니스 카드만 응답
```

### 9.2 클라이언트 역할

- 받은 데이터를 그대로 렌더링
- 위계별 **UI 스타일** 차이는 가능 (예: L1 대시보드는 읽기 전용 배지)
- 위계 강등 즉시 반영 위해 WebSocket/SSE 권한 변경 이벤트 구독

### 9.3 네비게이션 분기

| 네비 메뉴 | 노출 위계 |
|---|---|
| 조직 관리 | OA만 |
| 대시보드 (조직 전체) | OA, L1 |
| 대시보드 (팀 범위) | L1, L2 |
| 적합화 탭 | L2, L3, L4 |
| 칸반 | L2, L3, L4 |
| 파이프라인 | L2, L3, L4 |
| 설정 > 하네스 재설계 | L3만 |
| 설정 > 인프라 모드 | 모든 위계 (읽기 전용 표시) |
| 설정 > 감사 로그 | L3 이상 |
| 설정 > 데이터 내보내기 | L3, OA |

## 10. 위계 관련 감사 대상 (G-053)

아래 행위는 변경 불가 감사 로그 필수 (G-141 확장):

| 행위 | 기록 필드 |
|---|---|
| 권한 변경 | 변경자, 대상, 이전 위계, 이후 위계, 시각 |
| 조직 MUST 규칙 추가 | OA, 규칙 ID, 강제 수준, 시각 |
| 팀 생성/이동/삭제 | OA, 팀명, 이동된 팀원, 시각 |
| SSO 설정 변경 | OA, 프로토콜, 매핑 변경, 시각 |
| 인프라 모드 변경 | 슈퍼 어드민, 이전 모드, 이후 모드, 시각 |
| 하네스 재설계 요청 | L3, 이전 배정, 이후 배정, 시각 |
| 위계 강등으로 카드 재라우팅 | 원 담당자, 새 담당자, 카드 ID, 시각 |

## 11. 체크리스트 (코드 생성 시 자동 점검)

체인 실행 중 아래 항목 위반 감지되면 Conflict 자동 발동 (`91 § 1`):

- [ ] PM(L2)에게 기술 결정 카드가 라우팅되고 있지 않은가?
- [ ] 조직 MUST 규칙을 하위에서 해제하려 하고 있지 않은가?
- [ ] 하네스 재설계 UI가 L3 외에게 노출되고 있지 않은가?
- [ ] 에이전트 모델 직접 변경 드롭다운이 있지 않은가?
- [ ] 클라이언트 조건 분기로 데이터를 숨기려 하고 있지 않은가?
- [ ] OA만 수행 가능한 행위에 다른 위계 접근이 허용되지 않는가?
- [ ] 감사 로그 대상 행위가 `G-053` 목록에 포함되어 있는가?
- [ ] AiOPS 단독 판매 시 Wiring 위계를 요구하지 않는가?
