# 06_HITL — Human-In-The-Loop 설계 원칙

> 그릿지의 제품 철학: **"사람은 결정자, AI는 실행자."**
> 모든 사람의 승인 = 적합화 = 규칙 누적. 쌓일수록 AI가 정확해지는 구조.
> 규칙 ID: G-100~G-119.

---

## 1. 핵심 철학 (G-100, G-101)

### G-100 — 역할 분리

**사람 = 결정자 / AI = 실행자.** 이 구분은 UI/코드/문서 어디에서도 혼동되면 안 된다.

| 역할 | 할 수 있는 것 | 할 수 없는 것 |
|---|---|---|
| **사람** | 결정, 승인, 거부, 기각, 수정 요청 | 직접 코드 생성을 강요받지 않음 |
| **AI (Orchestrator)** | 제안, 추론, 호출 결정 | 사람 승인 없이 규칙 확정 불가 |
| **AI (Executor)** | 코드 생성, 검증, 파일 작성 | **추론 금지** (3 Orchestrator만 추론) |

**UI/문서에서 금지되는 표현:**
- ❌ "PM이 이 작업을 수행한다" (PM은 **결정**만)
- ❌ "AI가 이 규칙을 확정했다" (AI는 **제안**만, 확정은 사람)
- ❌ "에이전트가 자동으로 처리했다" (승인 없이 규칙 변경 불가)

### G-101 — 승인 = 적합화

사람이 AI 제안을 승인하는 **모든 행위**는 규칙으로 누적된다.

```
사람 승인 → spec-common.yaml 업데이트 → 다음 호출 시 AI가 규칙 준수
                ↓
            규칙 타임라인 기록 → 감사 로그 immutable
```

**6단계 자동 처리 (승인 버튼 클릭 시):**

1. 적합화 큐에서 카드 제거
2. 관련 아이템 진행 (HITL 해제)
3. `spec-common.yaml` 또는 `rules.md` 업데이트
4. 확정 규칙 타임라인에 기록 (오늘/어제/이번 주)
5. 감사 로그 immutable 기록 (G-141)
6. 토스트 알림 + 연관 카드 자동 갱신 (온톨로지 연쇄 추천 등)

**이 6단계 중 하나라도 누락된 UI는 HITL이 아니다.** 단순 버튼이다.

## 2. HITL 노드 4종 (G-102) — 핵심

### 2.1 카드 유형 매트릭스

| # | 유형 | 아이콘 | 좌측 바 | 담당 | 예 |
|---|---|---|---|---|---|
| 1 | **비즈니스 결정** | 🔶 | 주황 3px 실선 | L2 PM | "환불 시 포인트 복원?" |
| 2 | **기술 결정** | 🔷 | 파랑 3px 실선 | L3 기술 리드 | "낙관적 락 vs 비관적 락?" |
| 3 | **코드 패턴 승격** | 🔶 | 주황 3px 실선 | L3 (L4 제출) | "@Builder 패턴 4회 감지" |
| 4 | **온톨로지 추천** | 🔗 | 파랑 3px **점선** | L3 | "Facade 관련 규칙 2건, 92% 선택" |

**시각적 구분 원칙:**
- 비즈니스 = 주황 (정책 색)
- 기술 = 파랑 (공학 색)
- 점선 = "추천, 강제 아님" 신호

### 2.2 유형 1 — 비즈니스 결정 (🔶 policy_confirm)

**트리거:**
- SSOT Master 에이전트가 기획서 분석 중 정책 미정의 감지
- 기획서에 "MUST"만 쓰여 있고 구체 값이 없을 때
- 비교 대상 서비스 간 정책이 다를 때 (cafe24/sixshop/makeshop 비교)

**카드 표시 필수 요소:**
- 질문 (한 줄, 자연어)
- 근거 (참고 플랫폼 비교, 기획서 인용)
- 온톨로지 통계 ("87%가 X 선택" — 있으면)
- **"규칙으로 저장됩니다" 안내**
- 옵션 버튼 (최소 2~3개, 항상 "조건부 (직접 입력)" 포함)

**선택지 처리:**
- 버튼 클릭 → G-101 6단계 자동 처리
- "조건부" 선택 → 자연어 입력 → AI 구조화 → 확인 모달 → 규칙으로 저장

**예시:**
```
🔶 환불 시 사용된 포인트를 복원할까요?
   근거: 기획서 p.42 "MUST"만 명시. cafe24=복원, sixshop=미복원.
   [복원한다] [복원 안 한다] [조건부 (직접 입력)]
   → 선택 시 "환불 시 포인트 복원 (MUST)" 규칙 저장
```

### 2.3 유형 2 — 기술 결정 (🔷 tech_decision)

**트리거:**
- Tech Leader 에이전트가 spec-common 규칙에 근거해 기술 선택이 필요하다고 판단
- 트레이드오프가 있는 아키텍처 선택 (락 전략, 패턴 적용, 경계 분리 등)
- LucaPus 엔진이 "core vs domain" 분류에 확신이 없을 때

**카드 표시 필수 요소:**
- 질문 + `spec-common D-xxx` 참조
- 대안 비교 테이블 (장단점)
- AI 권장 + 신뢰도 (예: "87% 확신도로 A 권장")
- 관련 규칙 목록 (`relatedRules: ["rule-rbac", ...]`)
- 옵션 버튼 + "다른 방법 제안" (자연어 → 구조화)

**선택지 처리:**
- 확정 → 아이템 진행 + **온톨로지 연쇄 추천 카드 자동 생성** (유형 4로 전환)

**예시:**
```
🔷 PointUsage.consume()에 낙관적 락 vs 비관적 락?
   참조: spec-common D-025 (동시성 제어)
   대안:
     비관적 락 — DB 잠금. 안전/느림. TPS ↓
     낙관적 락 — 버전 충돌 재시도. 빠름/충돌 시 UX ↓
   AI 권장: 낙관적 락 (신뢰도 74%)
   관련: rule-redis-cache, rule-transaction
   [낙관적 락] [비관적 락] [다른 방법 제안]
```

### 2.4 유형 3 — 코드 패턴 승격 (🔶 code_pattern)

**트리거:**
- QA Verifier 에이전트가 **동일 수정 3회 이상** 감지
- L4 개발자가 코드 리뷰 중 같은 교정을 반복 (→ 자동 큐 제출)
- Claude Code가 `/gz-pattern` 으로 명시적 제안

**카드 표시 필수 요소:**
- 감지 근거 (수정 diff N건, 참조 아이템 ID 리스트)
- AI 규칙 초안 (자연어)
- 강제 수준 제안 (MUST/SHOULD/MAY)
- 옵션 4종: `[규칙으로 승격] [초안 수정 후 승격] [기각] [보류]`

**선택지 처리:**
- **승격:** G-101 6단계 + 다음 AI 호출부터 자동 적용
- **초안 수정:** 텍스트/카테고리/강제 수정 모달 → 수정 후 승격
- **기각:** 사유 입력 필수 + **제출한 개발자에게 알림**
- **보류:** 24시간 후 다시 큐에 표시

**패턴 감지 배지:**
```
🔶 @Builder 패턴을 엔티티에 적용
   ⚠ 패턴 감지 — 동일 수정 4회
   참조: PT-001, PT-002, PT-003, CP-001
   AI 초안: "모든 JPA 엔티티에 @Builder 적용" (MUST)
   [규칙으로 승격] [초안 수정] [기각] [보류]
```

### 2.5 유형 4 — 온톨로지 추천 (🔗 ontology_recommend)

**트리거:**
- 사용자가 다른 규칙을 확정한 직후 (연쇄 추천)
- 온보딩 시 코드베이스 감지 + 네트워크 관계 매핑
- 특정 도메인 진입 시 크로스 고객사 통계 기반

**카드 표시 필수 요소:**
- 트리거 규칙 ("Facade 패턴 확정으로 인한 추천")
- 추천 규칙 리스트 (체크박스, 일부 해제 가능)
- 각 규칙 네트워크 통계 ("92%가 선택")
- 출처 ("그릿지 네트워크 340개 프로젝트")
- 도메인 명시 ("Spring Boot + 이커머스")
- 옵션 3종: `[함께 확정] [개별 검토] [지금은 안 함]`

**시각적 구분:**
- 좌측 바 **파랑 점선** (실선은 강제, 점선은 "추천")

**예시:**
```
🔗 Facade 확정과 관련된 규칙 2건
   • Facade 내 @Transactional 필수 (92% 채택) [MUST]  ☑
   • Facade에서 DomainEvent 발행 (87% 채택) [SHOULD]  ☑
   출처: 그릿지 네트워크 340개 프로젝트 (Spring Boot + 이커머스)
   [함께 확정] [개별 검토] [지금은 안 함]
```

## 3. 위계별 라우팅 (G-103, G-104) — 03과 교차

### G-103 — 기술 결정 🔷 = L3/L4

- 기술 결정, 코드 패턴 승격, 온톨로지 추천 → **L3 기술 리드**에게 라우팅
- L4 개발자는 코드 패턴 **제출**만 가능, 승격은 L3
- PM(L2)에게 라우팅 **금지** — 감지 시 Conflict 자동 발동

### G-104 — 비즈니스 결정 🔶 = L2

- 정책/가격/UX 결정 → **L2 PM**에게 라우팅
- 전체 HITL의 ~10%만 PM에게 (나머지는 개발자)
- L3에게도 동일 카드가 보이지만 **PM 처리 대기 중**으로 표시 (L3는 개입 불가)

### 라우팅 알고리즘 (03 § 3.3 재확인)

```typescript
function routeDecision(node: HitlNode): UserLevel {
  switch (node.type) {
    case "policy_confirm":      return "L2";
    case "tech_decision":       return "L3";
    case "code_pattern":        return "L3";  // L4 제출 가능
    case "ontology_recommend":  return "L3";
    case "rule_conflict":       return "OA";  // 조직 간 충돌
  }
}
```

**원칙 위반 감지:** 코드에서 PM 위계에 `tech_decision` 라우팅 시도 → Conflict (`91 § 1`).

## 4. 온톨로지 추천 특수 규칙 (G-105)

### 4.1 자동 적용 금지

온톨로지 추천은 아무리 채택률이 높아도 **자동 적용 금지.**

- "92%가 선택" ≠ 자동 선택
- 고객이 직접 `[함께 확정]` 또는 `[개별 검토]` 또는 `[지금은 안 함]` 선택해야 규칙이 됨
- 사용자 수락/거부 이력은 설정 > 온톨로지 추천 현황에 보관

### 4.2 Mode B 특수 규칙

- Mode B (온프레미스) 고객의 적합화 데이터는 크로스 고객사 통계에 **포함 안 됨** (G-084)
- 단, Mode B 고객도 크로스 통계 **수혜는 받음** (다른 고객 익명 데이터 기반 추천)
- 크로스 통계 참여 opt-in 가능 (Mode B → A 효과, 고객 명시 동의 필요)

### 4.3 온톨로지 추천의 3가지 작동 시점

| 시점 | 트리거 | 카드 생성 경로 |
|---|---|---|
| **초기 온보딩** | 코드베이스 감지 + 관계 매핑 | `F-001-16` 온보딩 네트워크 추천 |
| **진행 중 (연쇄)** | 다른 규칙 확정 후 | 적합화 탭 `F-002-12` |
| **크로스 도메인** | 특정 모듈 진입 시 (예: 결제) | 적합화 탭 알림 |

## 5. HITL 병목 처리 (G-106)

### 5.1 병목 신호 감지

```
사람 노드에 ⚡ 2건 이상 대기
        ↓
파이프라인 뷰에 glow 효과 + 엣지 굵기 증가 + "⚡2건 대기" 라벨
        ↓ (4시간 초과)
Slack/이메일 알림 (해당 담당자에게)
        ↓ (24시간 초과)
에스컬레이션 — L3 → OA 보고
```

### 5.2 병목 해소 촉진 규칙

- 🔶 비즈니스 카드 — **모바일 Slack에서 인라인 승인 가능** (복잡한 건 웹 링크)
- 🔷 기술 카드 — CLI에서 `gridge adapt` 로 처리 가능 (단, 대안 비교가 복잡하면 웹 권장)
- 🔗 온톨로지 추천 — 긴급도 낮음 → "보류" 기본값, 병목 알림 대상 아님

### 5.3 Stage별 병목 영향

- Stage 0: HITL만 있음 → 병목 개념 없음
- Stage 1~2: AI 제안 대기 중, 병목 → 개발자 생산성 저하
- Stage 3: AI 풀 파이프라인 대기 → 병목 = 전체 파이프라인 멈춤

## 6. Stage별 HITL 동작 (G-107)

| Stage | HITL 범위 | 적합화 탭 | 칸반 | 파이프라인 |
|---|---|---|---|---|
| **0 모니터링** | 규칙 확인/수정만 | 동작 | 미표시 | 미표시 |
| **1 AI 보조** | 제안 검토 + 규칙 확정 | 동작 | 3컬럼 (제안/검토/반영) | 미표시 |
| **2 AI 협업** | 모듈별 AI 결과 검토 + 규칙 | 동작 | 6컬럼 (AI 모듈만) | AI 모듈만 |
| **3 AI 주도** | 풀 파이프라인 승인 | 동작 | 6컬럼 전체 | 전체 |

**불변 원칙:** Stage 0에서도 적합화(HITL 포함)는 **동작한다** (G-061). 적합화는 Stage와 무관한 제품 핵심.

## 7. CLI에서 HITL 처리 (G-108)

### 7.1 CLI 처리 가능 범위

```bash
$ gridge adapt

  🔶 결정 대기 2건
  🔷 결정 대기 3건
  🔶 코드 패턴 1건
  
  처리할 카드 선택: _
```

**가능한 처리:**
- 옵션 버튼식 선택 (기술 결정, 코드 패턴 승격)
- 자연어 "조건부" 입력 (비즈니스 결정)
- 빠른 기각/보류

### 7.2 웹으로 이관해야 하는 경우

복잡한 결정은 CLI에서 URL만 제공 + 웹에서 처리:

```
⚠ 이 결정은 대안 비교 + 온톨로지 추천이 복잡합니다.
  웹에서 처리 권장:
  https://app.gridge.ai/proj-001/adapt/adapt-003
```

**이관 트리거:**
- 대안이 3개 이상
- 관련 규칙 > 5개
- 온톨로지 연쇄 추천 카드 포함
- 첨부 파일 필요 (다이어그램, 스펙 문서)

### 7.3 웹-CLI 동기화

G-044 실시간 동기화: 어디서 처리하든 양쪽에 즉시 반영. 감사 로그는 처리 위치까지 기록.

## 8. HITL 감사 대상 (G-109)

모든 HITL 결정은 변경 불가 감사 로그 필수 (G-141):

```json
{
  "action": "적합화 결정",
  "actor": "이시니어(L3)",
  "nodeType": "tech_decision",
  "detail": "낙관적 락 → MUST 확정",
  "ref": "spec-common D-025",
  "selectedOption": "낙관적 락",
  "aiRecommendation": "낙관적 락 (74%)",
  "alignedWithAi": true,
  "at": "2026-04-14T10:33:00Z"
}
```

**추가 기록 대상:**
- `alignedWithAi` — 사람 선택이 AI 권장과 일치했는지 (학습 지표)
- `durationSec` — 카드 노출부터 결정까지 시간 (병목 분석)
- `conditionalInput` — "조건부" 선택 시 자연어 원문 (규칙 추출 재검증용)

## 9. HITL 승격 경로 (G-110)

패턴 감지 → 규칙 승격까지의 자동 파이프라인:

```
L4 개발자가 코드 리뷰 중 동일 수정 3회
        ↓
QA Verifier 자동 감지 + 🔶 코드 패턴 카드 생성
        ↓
L3 기술 리드 큐에 도착 → 옵션 4종 중 선택
        ↓ [규칙으로 승격] 선택 시
spec-common.yaml 업데이트 + 감사 로그
        ↓
다음 체인 실행 시 AI가 해당 규칙 자동 준수
        ↓ (다른 프로젝트/팀에서도 동일 패턴 감지 시)
조직 규칙으로 승격 후보 제안 (OA 승인 필요)
```

**원칙:** 자동 승격 없음. 각 단계에서 사람 승인 필수.
