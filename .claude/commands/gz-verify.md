# `/gz-verify <대상>` — 3중 검증 (V 체인)

## 목적
명세 문서 ↔ 레퍼런스 코드 ↔ 현재 구현을 3열로 대조. 사용자 결정 기록.

## 트리거
- **수동**: `/gz-verify <대상>` 입력
- **자동**: 사용자 요청에 "검증", "맞는지 확인", "레퍼런스와 비교", "대조" 키워드 감지 시

## 입력
```
대상 문서 (명세서/PDF/xlsx): <경로>
비교 대상 레퍼런스 코드: <경로>
현재 구현 위치 (있으면): <경로>  — 선택
```

## 실행
`90_execution_chain.md § V` 진입.

```
1. 사전 로드:
   - 대상 문서 전체 로드
   - 레퍼런스 코드 스캔
   - 현재 구현 스캔 (있을 경우)
2. 탐색: 대상 문서의 요구사항 목록 추출
3. 실행: 각 요구사항마다 레퍼런스/구현과 대조 → 일치/부분/불일치 판정
4. 검증: 불일치 건을 사용자에게 확인
5. 기록: 사용자 결정 → .context/issue/knowledge-*.md
6. 산출물: 3열 대조표 (Markdown 표)
```

## 출력
```
| 요구사항 | 명세서 | 레퍼런스 | 현재 구현 | 판정 |
|---|---|---|---|---|
| 트랜잭션 격리 | Serializable | Read Committed | Read Committed | 불일치 |
| 재시도 횟수 | 3회 | 3회 | 5회 | 불일치 |
| 타임아웃 | 10s | 10s | 10s | 일치 |
```

+ 불일치 건별 사용자 결정 로그.

## 예시
```
/gz-verify

 → 대상 문서: PRD-payment-v2.pdf
   레퍼런스 코드: ../reference-payment-service/
   현재 구현: src/payment/

[출력: 3열 대조표 + 5건 불일치 감지]

"트랜잭션 격리가 명세는 Serializable인데 구현은 Read Committed입니다.
 어떻게 할까요?
 [1] 구현 수정  [2] 명세 수정  [3] 예외 기록  [4] 보류"
```

## 금기
- 3개 입력 중 하나라도 없으면 실행 금지 (대상·레퍼런스는 필수, 현재 구현만 선택)
- 사용자 결정 없이 자동 수정 금지
- 판정을 "유사" 같은 모호한 표현으로 출력 금지 (일치/부분/불일치만)
