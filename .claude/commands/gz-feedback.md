# `/gz-feedback` — Feedback 3단계 대화형 입력

## 목적
룰북 자체의 개선 제안 (누락된 규칙, 모호한 정의, 실전에서 안 맞는 규칙 등)을 구조화 입력.

## 트리거
- **수동**: 사용자가 `/gz-feedback` 입력
- **자동**: 세션 종료 시 `/gz-self-feedback` 의 일부로 호출

## 입력
3단계 대화형:
```
1. 대상 파일 (어느 규칙 파일에 대한 피드백?)
2. 내용 (무엇을 바꾸고 싶은가?)
3. 발생 상황 (어떤 작업 중 느꼈는가?)
```

## 실행
`99_protocol.md § 3` 그대로. 세부:

```
1. 대상 파일 질문 (규칙 파일 경로 + 섹션)
2. 내용 질문 (자연어 설명)
3. 발생 상황 질문 (체인 / 작업 / 구체 케이스)
4. .context/issue/feedback-YYYY-MM-DD-HH-MM_<slug>.md 생성
5. 우선순위 추정 (사용 빈도 × 영향도)
6. 유사 피드백 스캔 (.context/issue/ 에 이미 있는지)
7. /gz-send-issue 권고
```

## 출력
```
피드백 파일 경로: .context/issue/feedback-*.md

분류:
  - 대상: <파일 경로>
  - 유형: 누락/모호/오류/개선
  - 우선순위: 높음/중간/낮음

다음 단계:
  [1] /gz-send-issue 로 GitHub Issue 전송
  [2] 로컬 보관 (팀 리뷰 전)
```

## 예시
```
/gz-feedback

Q1: 대상 파일은?
사용자: "06_hitl.md § 2.4 코드 패턴 노드"

Q2: 무엇을 바꾸고 싶나요?
사용자: "패턴 감지 임계치가 3회로 고정인데, 팀 규모에 따라 달라야 함. 작은 팀은 2회여도 충분"

Q3: 어떤 상황에서 느꼈나요?
사용자: "3명 팀 프로젝트에서 2회 반복되는 패턴이 유의미한데 승격 후보로 안 잡힘"

→ 파일 생성: feedback-2026-04-18-15-00_pattern-threshold-team-size.md
  우선순위: 중간 (팀 규모별 커스터마이징 요구)
```

## 금기
- 3단계 중 하나라도 빠지면 파일 생성 금지 (구조화가 핵심)
- 사용자 감정적 표현을 그대로 파일에 기록 금지 (건설적 표현으로 정리)
- 팀 비방/인신공격 내용 기록 금지 (룰북 개선에 집중)

## 참조

- `/gz-send-issue` GitHub Issue 전송
- `/gz-self-feedback` 세션 종료 통합 실행
