## PW-009-03 — 각 단계 상세 (MUST)

### R1 근거 수집

- 플랫폼별 수집 진행 리스트 (cafe24 / sixshop / makeshop 등)
- 각 플랫폼마다 "수집 완료 N개 / 대기 N개" 배지
- 완료 시 자동 R2 진행

### R2 관리 범위

- Feature → Sub-feature 트리 (접이식)
- 각 노드 옆 아이콘: 🔸 기본 / 🔶 PM 결정 필요 / 🔷 기술 결정 필요
- 우선순위 (high/medium/low) 태그

### R3 비교 분석

- 3열 비교 테이블 (플랫폼 × 정책)
- 차이점 하이라이트 (배경 주황)
- 사용자가 "이 플랫폼 정책 참고" 선택 가능 → R6에 반영

### R4 시나리오 (HITL ✅ PM)

- BDD 시나리오 카드 형식 (Given / When / Then)
- ⚡ 미정 → [적합화에서 결정] 링크 (Wiring 적합화 탭으로 이동)
- PM 결정 대기 카드 4개까지 표시, 5개+ 는 "N개 더" 버튼

예시 카드:
```
🔶 환불 시 포인트 복원 여부
  Given: 사용자가 주문 완료
  When: 환불 요청
  Then: 포인트를 복원한다 / 복원 안 한다 / 조건부?
  [결정하기 →]
```

### R5 데이터 설계 (HITL ✅ L3)

- 엔티티 카드 그리드 (User / Order / Product / ...)
- 관계도 (엔티티 간 `hasMany` / `belongsTo` / ...)
- ⚡ 기술 결정 → [적합화에서] 링크
- L3만 엔티티 클릭 시 필드 상세 / 락 전략 / 인덱스 보임

### R6 규칙 정리

- MUST / SHOULD / MAY 그룹 세로 3열
- 각 규칙 카드: 규칙명 / 소스 (R3 근거 / R4 시나리오 등) / 영향 범위
- L3 이상은 각 규칙 옆 "규칙으로 승격" 버튼 (온톨로지 연쇄 추천 트리거)

### R7 문서 생성

- 생성된 문서 목록 (6~8개):
  - `spec-common.yaml`
  - `architecture.md`
  - `entities.md`
  - `api-contracts.md`
  - `test-scenarios.md`
  - `migration-plan.md`
  - `Feature Kit (기능별)`
  - `Core Baseset`
- 각 문서 옆 [미리보기] (PM 언어 렌더링) / [원본 다운로드] (L3+ 만 코드)

## PW-009-04 — 진행 상태 시각화 (MUST)

각 단계 노드는 상태 표시:

| 상태 | 시각 | 조건 |
|---|---|---|
| 대기 | 회색 외곽선 | 이전 단계 미완료 |
| 진행 중 | 파란 glow + pulse | 현재 단계 |
| 완료 | 초록 체크 | 완료 |
| HITL 대기 | 주황 ⚡ + glow | R4/R5에서 미정 발생 |
| 에러 | 빨강 ✗ | 분석 실패 |

### 순차 진행 애니메이션 (SHOULD)

데모 시나리오용 (SIM-005):

```tsx
// [분석 시작] 버튼 클릭 시
for (const step of steps) {
  await simulate(step, 1500);  // 1.5초 간격
  updateStatus(step, 'active');
  await sleep(800);
  updateStatus(step, 'complete');
}
```

## PW-009-05 — HITL 연결 (MUST)

R4 (PM) / R5 (L3) 에서 발생한 HITL은 적합화 탭의 "결정 필요" 통합 리스트에 자동 등록.

### 데이터 플로우

```
R4에서 PM 결정 필요 감지
  ↓
useSpecStore.addHITL({ type: 'business', step: 'R4', ... })
  ↓
useAdaptStore 에도 동일 카드 추가 (cross-store sync)
  ↓
적합화 탭 / 파이프라인 (사람 노드) 양쪽에 반영
```

### 링크 동작

R4 카드의 [결정하기 →] 클릭:
- 적합화 탭 이동 + 해당 카드로 스크롤 + highlight

## PW-009-06 — 순서 변경 금지 (MUST)

**G-028 절대 불허 원칙.** UI에서도 이를 시각적으로 강제:

- 노드 수동 이동 불가 (드래그 비활성)
- R1 건너뛰고 R3 먼저 실행 불가 (비활성 상태)
- 순서 바꾸려는 버튼 / 메뉴 **존재 금지**

위반 감지 시 Conflict:
```
❌ R4 → R1 역방향 처리 감지
❌ R2 건너뛰고 R3 실행
❌ 순서 변경 버튼 UI 발견
```
